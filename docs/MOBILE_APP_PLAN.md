# Mobile App Plan — Tauri iOS & Android

## Overview

Extend the existing Tauri v2 desktop app to also target iOS and Android, sharing
100% of the TypeScript frontend. The interesting design question is not "how do
we build a phone app" — Tauri's CLI handles most of that — but "what does Orbit
look like on a phone, given that the OS now provides an on-device LLM for free."

This document captures the architectural decisions for that exploration. It is
**a design record, not an implementation checklist.** Phases below describe the
shape of the work but are deliberately not estimated.

## Why Tauri (and not a rewrite)

We already ship the same TypeScript source to both web (Cloudflare Pages) and
desktop (Tauri v2). Adding mobile preserves that single-source story:

| Path                  | Frontend       | Backend            | Distribution            |
|---|---|---|---|
| Web                   | Vite → CF Pages| CF Functions proxy | URL                     |
| Desktop (current)     | Vite → Tauri   | Rust (`src-tauri/`)| GitHub Releases         |
| **iOS / Android (new)** | Vite → Tauri | Rust + native plugin | App Store / Play Store |

The alternatives (React Native, Flutter, Capacitor + framework rewrite) all
require porting the WebGL globe, MapLibre integration, custom Earth tile layer,
HLS pipeline, and Orbit chat UI to a new runtime. Tauri keeps all of that
unchanged.

Tauri v2 mobile is still marked beta. We accept that risk in exchange for the
single-source benefit; the fallback if it doesn't work out is web-on-mobile via
PWA install.

## Orbit on Mobile — The Interesting Bit

The desktop version of Orbit supports any OpenAI-compatible endpoint, with a
local-LLM story for users running Ollama / LM Studio / llama.cpp on the same
machine. That model breaks on a phone for two reasons:

1. iOS App Transport Security blocks plain HTTP by default; carving out
   `localhost` exceptions is awkward and the use case (a phone talking to a
   nearby llama.cpp on a laptop) is niche.
2. Phones now ship their **own** on-device LLM. Apple Intelligence has been
   exposed to third-party apps via the **Foundation Models framework** since
   iOS 26 (WWDC 2025, GA fall 2025). It is free, streams, and supports tool
   calling.

The mobile build therefore drops the localhost path entirely and adds Apple
Intelligence as a first-class provider, with HTTPS cloud endpoints as fallback.

### Provider matrix (mobile)

| Device                                  | Default provider              | Fallback         |
|---|---|---|
| iPhone 15 Pro / 16 / 17 / M-series iPad | Apple Intelligence (on-device)| HTTPS cloud      |
| Older iPhone / iPad                     | HTTPS cloud                   | Local engine     |
| Android (Gemini Nano-capable)           | Gemini Nano (Phase 4+)        | HTTPS cloud      |
| Android (other)                         | HTTPS cloud                   | Local engine     |

The local keyword-engine fallback in `docentEngine.ts` keeps the app
fully-offline-capable even with no network and no on-device model.

### Why on-device is the right default on iPhone

- **Free.** No API cost, no per-user metering, no infra.
- **Private.** Questions never leave the device — pairs perfectly with our
  offline-dataset story.
- **Offline.** Works in airplane mode, in-flight, in remote field use.
- **Zero binary cost.** The model ships with the OS — our `.ipa` doesn't grow.
- **Streaming + tool calling are first-class** in the Foundation Models Swift
  API.

### The hard constraint: 4,096-token context window

Apple's on-device model has a **4k total** context budget — input tokens plus
output tokens, in the same session. iOS 26.4 (March 2026) added context-window
management helpers (TN3193) but did not expand the budget.

Our current `docentContext.buildSystemPromptForTurn()` packs the entire dataset
catalog into the system prompt on Turn 0 and a compact catalog on Turn ≥1.
Neither fits in 4k once you add conversation history and a model reply.

**Architectural change forced by this:** the catalog becomes a tool the model
calls instead of a block of text in the prompt. We apply this to **both** the
cloud and on-device paths so the two providers share one retrieval shape — the
cloud path benefits independently from smaller token bills and a simpler
`docentContext.ts`.

### Catalog-as-tool (instead of catalog-in-prompt)

Today (cloud path):

```
SYSTEM: <persona> + <full catalog> + <tool defs>
USER:   "what's a good ocean dataset?"
LLM:    "Try INTERNAL_SST_ANOMALY <<LOAD:INTERNAL_SST_ANOMALY>>"
```

Proposed (both cloud and on-device):

```
SYSTEM: <persona> + <tool defs only>     // ~300 tokens, not thousands
USER:   "what's a good ocean dataset?"
LLM:    [tool call] search_catalog(query: "ocean")
TOOL:   [5 dataset summaries returned]
LLM:    [tool call] load_dataset(id: "INTERNAL_SST_ANOMALY")
LLM:    "Loading the sea-surface temperature anomaly dataset..."
```

The full catalog never enters the context window. Only the ~5 candidates the
model asked for do, and only when it asks. On the cloud path this lets us
delete the turn-aware prompt-size juggling in `docentContext.ts` and
materially shrinks per-turn token usage. On the on-device path it is the only
way the architecture fits in 4k tokens at all. Same retrieval shape, two
providers — that's the point.

### Guided generation > inline `<<LOAD:...>>` markers

Today `docentService.extractActionsFromText()` regex-parses inline markers
because LLMs are unreliable about tool-call formatting. Foundation Models'
**guided generation** constrains decoder output to a Swift `@Generable`
struct — the model literally cannot produce malformed output. So:

```swift
@Generable
struct DocentTurn {
  @Guide(description: "What to say to the user")
  let text: String

  @Guide(description: "Optional dataset ID to load inline")
  let datasetToLoad: String?
}
```

The marker-parsing fallback in `docentService.ts` stays as a safety net for the
cloud path but becomes dead code on the on-device path.

## The `apple-intelligence` Tauri Plugin

A new mobile Tauri plugin under `src-tauri/plugins/apple-intelligence/`. This
will be (as far as we can tell) the first Tauri plugin for Foundation Models.

### Swift surface

```swift
import FoundationModels
import Tauri

class AppleIntelligencePlugin: Plugin {
  private var session: LanguageModelSession?

  @objc public func isAvailable(_ invoke: Invoke) {
    let model = SystemLanguageModel.default
    invoke.resolve([
      "available": model.availability == .available,
      "reason": String(describing: model.availability)
    ])
  }

  @objc public func chatStream(_ invoke: Invoke) throws {
    // 1. Parse messages + tool definitions from invoke args
    // 2. Build LanguageModelSession with our catalog-search tool
    //    and load_dataset tool, both as @Generable structs
    // 3. Stream the response, emit `delta` / `tool-call` / `done`
    //    events back to JS via trigger("orbit-stream", ...)
    // 4. Handle .exceededContextWindowSize by starting a new
    //    session with a compacted history (TN3193 pattern)
  }
}
```

### Rust shim

~30 lines: `tauri::plugin::Builder::new("apple-intelligence")` registering
two commands (`is_available`, `chat_stream`) that delegate to the Swift class
via Tauri's mobile bridge. `#[cfg(target_os = "ios")]` so it compiles to a
no-op on other platforms.

### JS provider

A new `AppleIntelligenceProvider` class in `src/services/llmProvider.ts` that
satisfies the same async-generator interface as the existing OpenAI-compatible
provider. `docentService.processMessage()` does not change — it already treats
the LLM as an abstract stream source.

### Settings UI

`chatUI.ts` settings form gains a "Local (Apple Intelligence)" option in the
provider dropdown, only shown when the plugin's `isAvailable()` returns true.
The API key field is hidden for that selection.

## Project-Specific Mobile Adaptations

Items flagged from a code walkthrough of `src-tauri/` and `src/`:

### Rust modules

| Module | Mobile status | Action |
|---|---|---|
| `tile_cache.rs` | Compiles, paths resolve correctly | None — works as-is |
| `download_manager.rs` | Compiles | None — works as-is |
| `download_commands.rs` | Compiles | None — works as-is |
| `keychain.rs` | iOS: works (`apple-native`). Android: no backend | Gate Android with `cfg`, use `EncryptedSharedPreferences` via JNI later |
| `main.rs` updater registration | Updater plugin not supported on mobile | Wrap in `#[cfg(desktop)]` |

### Capabilities

`src-tauri/capabilities/default.json` allows `http://localhost:*`,
`http://*:11434`, `http://*:1234`, `http://*:8080` for desktop local-LLM
support. **Mobile gets a separate capability file** (`mobile.json`) with only:

- `https://*` (NASA GIBS, Vimeo proxy, cloud LLM providers)
- Asset protocol scoped to app data (already configured)

No cleartext HTTP. No ATS exceptions. No `usesCleartextTraffic` on Android.

### Frontend libraries

| Library | Mobile status | Risk |
|---|---|---|
| MapLibre GL JS | Works in WKWebView and Android WebView | **Performance unknown on mid-tier Android** — profile early |
| `earthTileLayer.ts` custom WebGL | Same as MapLibre | Same |
| HLS.js | iOS: passes through to native `<video>`. Android: full HLS.js | None |
| Vimeo proxy flow in `hlsService.ts` | HTTPS, works | None |
| Asset protocol (`convertFileSrc()`) | Supported on both platforms | Validate during Phase 1 spike |

### UI

`chatUI.ts` and `playbackController.ts` already handle the ≤600px portrait
breakpoint, which gives mobile a head start. New work needed:

- `IS_MOBILE_NATIVE` runtime flag (distinct from `IS_TAURI`) using
  `@tauri-apps/api/app` platform probe
- Safe-area insets (notch, home indicator, status bar)
- Bottom-sheet pattern for the dataset browser instead of side panels
- Larger touch targets in the chat input and playback transport
- Native share sheet for "share this dataset"

## CI Strategy

**New workflow `.github/workflows/mobile.yml`:**

- `macos-14` (or latest with Xcode 16+) for iOS — Xcode and iOS SDKs
  pre-installed on GitHub-hosted runners
- `ubuntu-latest` for Android — Android SDK pre-installed
- PR-only compile checks with `--debug`, no signing, no artifacts
- Cost-conscious: macOS runner minutes are ~10× Linux. iOS only runs on PRs
  that touch `src-tauri/`, `src/`, or the workflow itself

**Extend `release.yml`:**

- On version tag: build signed IPA + AAB
- Upload IPA to TestFlight via fastlane
- Upload AAB to Play Console Internal Testing track
- Manual promotion from Internal → Production after smoke testing

## Phased Plan

**Phase 1 — Spike (this branch's follow-up):**

1. `tauri ios init` and `tauri android init`; commit `src-tauri/gen/`
2. Add iOS/Android sections to `tauri.conf.json` (bundle IDs, orientation, min SDK)
3. Gate updater plugin and `keychain.rs` Android path with `cfg`
4. Create `src-tauri/capabilities/mobile.json` (HTTPS only)
5. Add `dev:ios` / `dev:android` / `build:ios` / `build:android` scripts
6. Manual smoke test on simulator and a real device (existing iPhone 15 Pro+)
7. Validate: globe renders, datasets load, asset protocol works

**Phase 2 — CI compile checks:**

8. New `.github/workflows/mobile.yml` with debug compile on PR
9. Document the manual dev loop in the project README

**Phase 3 — Catalog-as-tool refactor (cloud + on-device):**

10. Refactor `docentContext.ts`: replace prompt-stuffed catalog with a
    `search_catalog(query, limit)` tool the model calls when it needs
    recommendations. Drop `buildSystemPromptForTurn()`'s turn-awareness — the
    system prompt becomes static and small (~300 tokens) regardless of turn.
11. Implement the catalog search tool itself in `docentService.ts`. Simple
    keyword + category match against the in-memory catalog from `dataService.ts`
    is enough for v1; we already have similar logic in `docentEngine.ts` we
    can lift.
12. Update the OpenAI-compatible provider in `llmProvider.ts` to surface the
    new tool via OpenAI-style JSON-schema function definitions.
13. Update `extractActionsFromText()` and stream chunk handling: tool-call
    results become first-class, the inline `<<LOAD:...>>` marker stays as a
    fallback for providers that don't honor function calling.
14. Test: verify cloud path quality on the existing test cases and watch token
    usage drop. Catalog data is no longer in every system prompt.

**Phase 4 — Apple Intelligence integration:**

15. Build `tauri-plugin-apple-intelligence` (Swift + Rust shim)
16. Add `AppleIntelligenceProvider` to `llmProvider.ts`. Reuses the
    catalog-as-tool design from Phase 3 — same tool surface, same
    `docentService` flow, just a different transport.
17. Define the on-device `@Generable` `DocentTurn` and tool structs in Swift,
    mirroring the JSON-schema definitions used by the cloud provider.
18. Settings UI: provider dropdown gains "Local (Apple Intelligence)" option
19. Read and comply with the Foundation Models acceptable-use requirements

**Phase 5 — Mobile UX polish:**

20. `IS_MOBILE_NATIVE` flag, safe-area insets, bottom-sheet browser
21. Native share sheet, deep linking (`zyra://dataset/...`)
22. Profile MapLibre on mid-tier Android; decide on a 2D fallback if needed

**Phase 6 — Signed builds + store distribution:**

23. Extend `release.yml` for signed IPA + AAB
24. TestFlight pipeline; first invited testers
25. Play Console Internal Testing pipeline
26. Submit for App Store and Play review
27. `docs/RELEASE_MOBILE.md` runbook

**Phase 7 (deferred) — Android on-device LLM:**

28. Mirror the Apple Intelligence plugin for Gemini Nano via AICore / ML Kit
    GenAI APIs
29. Same provider interface, same retrieval-based catalog approach (Phase 3
    work means there is nothing further to refactor on the JS side)

## Open Questions / Decisions Captured

1. **Local-HTTP LLM path on mobile**: dropped. iPhone uses Apple Intelligence;
   iPad uses Apple Intelligence (M-series iPads support it); cloud HTTPS as
   fallback on all devices. Simplifies the capability file and ATS story.

2. **Catalog-as-tool refactor scope**: applied to **both** the cloud and
   on-device paths in Phase 3, before the Apple Intelligence integration
   lands. Doing the cloud refactor first means the on-device provider can
   slot into an already-proven retrieval architecture instead of being its
   own special case. Side benefits on the cloud path: smaller per-turn token
   bills, deletion of `buildSystemPromptForTurn()`'s turn-awareness, and one
   fewer code path to maintain. The inline `<<LOAD:...>>` marker fallback
   stays as a safety net for any provider that doesn't honor function
   calling.

3. **Android first vs iOS first**: iOS first. We have an existing developer
   account, the on-device LLM story is more compelling, and Apple's framework
   ergonomics are better understood. Android can follow once the architecture
   is proven.

4. **Bundled local model (Core ML / MLX / llama.cpp)**: not in scope. Apple
   Intelligence covers our supported devices. Bundling would add 1–4 GB to
   the `.ipa`, exceed App Store cellular caps, and complicate updates. We
   revisit only if Foundation Models proves insufficient in real use.
