§9 · Publishing & search

Publish datasets. Build tours. Index for search.

Publish once · seen on every TerraViz install. The publisher portal at /publish is live today behind Cloudflare Access — a single-form workflow for dataset entry, tour authoring, asset upload, and semantic-search indexing. The companion terraviz CLI hits the same publisher API: terraviz publish dataset.yaml, scriptable from any CI for research orgs running scheduled pipelines. Both paths run through the same edge-compute + SQL DB + object-storage backend that powers the rest of the catalog stack — same edge, same auth, no separate moving parts. Once cross-node federation lands, a subscribing peer's catalog will pull your row on its next sync; no separate distribution, no CDN hand-off, no museum partnership required. The Cloudflare-flavoured names below (Pages Functions, D1, R2, GitHub Actions transcode, Images, Vectorize, Workers AI, Access) are the today-deploy; §8 covers the portability story for operators wanting a different cloud. Full spec in CATALOG_PUBLISHING_TOOLS.md.

Publisher portal

A single-form workflow at /publish behind staff SSO (Cloudflare Access today). Title, slug, abstract, asset upload, thumbnail, captions, time range, categories, license, visibility. Preview opens the app in a new tab against the draft row — same renderer, same playback, same chat — so publishers see exactly what visitors will see before pressing Publish. Lazy-loaded the same way Three.js is; non-publisher visitors never pay the bundle cost.

Tour creator

Capture-mode UI: hit Record, fly the camera, swap layouts, drop in captions, and the recorder emits flyTo / loadDataset / setEnvView / showRect / pauseForInput tasks into a tour.json the existing engine plays back identically. Same format Science On a Sphere's tour-builder produces — tours authored elsewhere import unchanged.

Asset & transcode pipeline

A publisher's MP4 — or a stack of individual frames — uploads straight to R2 over a presigned PUT; a GitHub Actions runner re-encodes it with ffmpeg into a 4K / 1080p / 720p HLS ladder, writes the segments back to R2, and the row's data_ref flips atomically. Cloudflare Stream was evaluated and dropped — its standard-plan 1080p ceiling was too low for 4K spherical video. Sphere thumbnails auto-generate at upload time as 2:1 equirectangular crops; Cloudflare Images can derive resolution variants for image datasets. Content-addressed R2 keys so revisions never invalidate caches. Full flow lives in CATALOG_ASSETS_PIPELINE.md.

Authoring CLI

Alternative to the portal for research orgs running scheduled visualization pipelines, CI jobs, or anyone who'd rather script than click. Hits the same publisher API as the portal, with server-side validation as the source of truth. Distributed as signed standalone binaries plus an npm package; verification keys checked into the repo so downstreams can audit the supply chain.

Semantic search

Cloudflare Vectorize stores a 768-dim embedding per dataset. The public /api/v1/search endpoint runs Vectorize semantic search (with a text fallback) today, and the docent's search_datasets LLM tool queries the same index (used by Orbit when a visitor asks "show me something about hurricanes"). Once federation lands, peer rows reindex locally on sync, so each node owns its own search experience without phoning home.

The publish path

A dataset goes through five steps from a publisher's upload to a visitor's search hit. Each step is owned by a different cloud primitive (Cloudflare today — see §8 for the portability story); the whole chain runs on the same edge.

Where it stands

The portal, CLI, asset pipeline, and search index are live; cross-node federation and the finer-grained publishing controls land in later phases. Mirrors §8's federation cadence — work the model first, then the auth, then the UI.
