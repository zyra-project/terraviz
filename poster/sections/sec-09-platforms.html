§6 · One codebase, every platform

One TypeScript codebase. Five places it runs.

The same source tree ships as a web app on Cloudflare Pages, a desktop app on Windows / macOS / Linux, and an iOS / Android mobile app. Tauri v2 wraps the app in a native shell and exposes platform-specific capabilities (offline tile cache, OS keychain, local-LLM HTTP allowlist) behind a single runtime gate — window.__TAURI__ — so web builds tree-shake every native code path away. Mobile is additive on top of the existing desktop tree; src-tauri/gen/apple/ and src-tauri/gen/android/ are the generated host projects.

Branching at one runtime gate

The app decides at runtime whether it's running in a browser, a desktop window, or a phone app — and lazy-loads the right capability shim. Web builds don't even download the Tauri-only modules.

What's next

Orbit on-device, where the OS already ships an LLM

Mobile is the most interesting deployment for the docent. Both Apple and Google now ship system-level LLMs — Apple Intelligence Foundation Models on iOS (≥ 18.1 on supported hardware) and Gemini Nano via AICore on Android. Bridging Orbit into those means the chat is free, private, and offline-by-default on eligible devices, with no API key, no provider relationship, no network round-trip.

The shape is a small Tauri plugin per platform — Swift wrapping FoundationModels, Kotlin wrapping AICore on Android — surfaced behind the existing llmProvider interface. The macOS desktop build already wires Apple's Foundation Models through this provider; extending it to iOS and Android phones is the forward direction. On hardware too old for system LLMs, the same provider gracefully falls back to the OpenAI-compatible path that web and desktop already use. Mobile on-device Orbit is not yet shipped.
